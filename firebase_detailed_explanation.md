# Comprehensive Guide to Firebase Integration in NearNeed

NearNeed relies on **Google Firebase** as its complete backend infrastructure. This eliminates the need to build, host, and maintain custom servers. Specifically, NearNeed leverages two core Firebase products: **Firebase Authentication** (for secure, passwordless login) and **Cloud Firestore** (for real-time, NoSQL database management).

This document provides an exhaustive breakdown of how authentication is handled, how the database is structured, and the exact commands used to read, write, and sync data in real-time.

---

## 1. Firebase Authentication: The Phone OTP Flow

In modern gig-economy apps, asking users to remember passwords leads to high drop-off rates. NearNeed uses **Phone Number Verification** to securely authenticate users via SMS OTP (One-Time Password).

### 1.1 Triggering the OTP
When the user enters their phone number on the login screen, we trigger the `PhoneAuthProvider`.

```java
// 1. Configure the verification options
PhoneAuthOptions options = PhoneAuthOptions.newBuilder(FirebaseAuth.getInstance())
    .setPhoneNumber("+91" + phoneNumber)       // The phone number to verify
    .setTimeout(60L, TimeUnit.SECONDS)         // Timeout duration for the SMS
    .setActivity(this)                         // The current Android Activity
    .setCallbacks(mCallbacks)                  // The listener that handles the result
    .build();

// 2. Send the request to Google's servers
PhoneAuthProvider.verifyPhoneNumber(options);
```

### 1.2 Handling the Callbacks
Firebase handles the actual SMS sending. Our app must listen for the result using `PhoneAuthProvider.OnVerificationStateChangedCallbacks`.

```java
private PhoneAuthProvider.OnVerificationStateChangedCallbacks mCallbacks = 
    new PhoneAuthProvider.OnVerificationStateChangedCallbacks() {

    @Override
    public void onVerificationCompleted(PhoneAuthCredential credential) {
        // This is triggered if the phone auto-reads the SMS via Google Play Services.
        // It bypasses the need for the user to manually type the OTP!
        signInWithPhoneAuthCredential(credential);
    }

    @Override
    public void onVerificationFailed(FirebaseException e) {
        // Triggered if the phone number is invalid or quota is exceeded.
        Toast.makeText(context, "Verification Failed: " + e.getMessage(), Toast.LENGTH_LONG).show();
    }

    @Override
    public void onCodeSent(String verificationId, PhoneAuthProvider.ForceResendingToken token) {
        // Triggered when the SMS is successfully dispatched.
        // We save the verificationId to compare against what the user types in the next screen.
        storedVerificationId = verificationId;
        Intent intent = new Intent(this, OtpVerifyActivity.class);
        startActivity(intent);
    }
};
```

### 1.3 Signing the User In
When the user types the 6-digit code, we combine it with the `storedVerificationId` to create a `Credential`, and then sign them in:

```java
PhoneAuthCredential credential = PhoneAuthProvider.getCredential(storedVerificationId, userEnteredOtp);

FirebaseAuth.getInstance().signInWithCredential(credential)
    .addOnCompleteListener(this, task -> {
        if (task.isSuccessful()) {
            FirebaseUser user = task.getResult().getUser();
            String uniqueUserId = user.getUid(); // The unique, permanent ID for this user
            // Proceed to the app dashboard...
        } else {
            Toast.makeText(this, "Incorrect OTP", Toast.LENGTH_SHORT).show();
        }
    });
```

## 2. Cloud Firestore Architecture: The NoSQL Paradigm

Unlike traditional SQL databases (like MySQL) that use rigid Tables and Rows, Cloud Firestore is a **NoSQL Document Database**. 
- **Collections**: Equivalent to SQL Tables. They hold multiple documents.
- **Documents**: Equivalent to SQL Rows. They are JSON-like objects that hold key-value pairs (Strings, Numbers, Arrays, Booleans).

### 2.1 NearNeed's Data Structure
The app utilizes three primary collections at its root level:
1. **`users`**: Stores profile data. The Document ID is strictly the Firebase Authentication UID to strictly link a login to a profile.
2. **`bookings`**: Stores job requests. The Document IDs are auto-generated by Firebase.
3. **`chats`**: Stores conversations. Contains a sub-collection of `messages` to keep the database flat and highly performant.

## 3. Firestore CRUD Operations: Writing Data

Writing data to Firestore can be done in three ways depending on the goal: creating a new record, setting a specific record, or updating an existing one.

### 3.1 Creating a New Booking (`.add()`)
When a Seeker books a Provider, we need a brand new document with a unique ID. We use the `.add()` method.

```java
FirebaseFirestore db = FirebaseFirestore.getInstance();

// 1. Build the data payload as a HashMap
Map<String, Object> bookingData = new HashMap<>();
bookingData.put("seekerId", "USER_UID_123");
bookingData.put("providerId", "PROVIDER_UID_456");
bookingData.put("serviceName", "AC Repair");
bookingData.put("status", "pending");

// Use FieldValue.serverTimestamp() so the time is determined by Google's servers,
// preventing issues if the user's phone clock is wrong.
bookingData.put("timestamp", FieldValue.serverTimestamp()); 

// 2. Add it to the "bookings" collection
db.collection("bookings")
    .add(bookingData)
    .addOnSuccessListener(documentReference -> {
        // Firebase automatically generated a random string ID for this booking!
        String newBookingId = documentReference.getId(); 
        Log.d("Firestore", "Booking created with ID: " + newBookingId);
    });
```

### 3.2 Saving a User Profile (`.set()`)
When a user signs up, we DO NOT want a random ID. We want their Firestore Document ID to exactly match their Authentication UID. We use `.set()`.

```java
Map<String, Object> userData = new HashMap<>();
userData.put("fullName", "John Doe");
userData.put("phone", "+919876543210");
userData.put("accountType", "Seeker");

// Notice we use .document(uid).set() instead of .add()
db.collection("users").document(firebaseAuthUid)
    .set(userData)
    .addOnSuccessListener(aVoid -> {
        // Profile securely linked to login credentials
    });
```

### 3.3 Editing a Profile (`.update()`)
If a Provider wants to add a bio to their profile, we CANNOT use `.set()`. Using `.set()` completely overwrites the document, deleting their name and phone number. Instead, we use `.update()`, which only modifies the specified keys.

```java
Map<String, Object> updates = new HashMap<>();
updates.put("bio", "I am a certified electrician with 5 years of experience.");
updates.put("isVerified", true);

db.collection("users").document(firebaseAuthUid)
    .update(updates)
    .addOnSuccessListener(aVoid -> {
        // Bio added, but fullName and phone are completely untouched!
    });
```

## 4. Firestore CRUD Operations: Reading Data

### 4.1 Fetching a Single Document (Profile View)
When navigating to the Profile Screen, we need to load the user's data once.

```java
db.collection("users").document(currentUserId)
    .get()
    .addOnSuccessListener(documentSnapshot -> {
        if (documentSnapshot.exists()) {
            // Data extraction
            String name = documentSnapshot.getString("fullName");
            
            // Arrays in Firestore are returned as Java Lists
            List<String> categories = (List<String>) documentSnapshot.get("categories");
            
            updateUI(name, categories);
        }
    });
```

### 4.2 Querying Multiple Documents (The Bookings Tab)
When a Provider opens their dashboard, we must query the database to find ALL bookings where they are the assigned provider. This requires filtering and sorting.

```java
db.collection("bookings")
    .whereEqualTo("providerId", currentUserId)  // Filter: Only my jobs
    .whereEqualTo("status", "pending")          // Filter: Only pending jobs
    .orderBy("timestamp", Query.Direction.DESCENDING) // Sort: Newest first
    .get()
    .addOnSuccessListener(querySnapshot -> {
        
        // Loop through all documents that matched the query
        for (QueryDocumentSnapshot doc : querySnapshot) {
            
            // MAGIC: Firestore can automatically convert the JSON document 
            // directly into a custom Java Object (POJO) using Reflection!
            Booking booking = doc.toObject(Booking.class);
            
            // We manually attach the Document ID so we can update it later if needed
            booking.setBookingId(doc.getId()); 
            
            bookingList.add(booking);
        }
        recyclerViewAdapter.notifyDataSetChanged();
    });
```

## 5. Real-Time Syncing: The Power of Snapshot Listeners

The most powerful feature of Firestore is its ability to push updates to the app instantly without the app having to refresh or poll the server. NearNeed uses this extensively for the **Chat System**.

Instead of using `.get()`, we use `.addSnapshotListener()`.

```java
// We listen to the "messages" sub-collection inside a specific chat document
db.collection("chats").document(chatId).collection("messages")
    .orderBy("timestamp", Query.Direction.ASCENDING)
    .addSnapshotListener((snapshots, error) -> {
        if (error != null) { return; }

        // snapshots.getDocumentChanges() returns ONLY the documents that have 
        // changed since the last time this block ran.
        for (DocumentChange dc : snapshots.getDocumentChanges()) {
            
            // If a new message was added to the database by the other user...
            if (dc.getType() == DocumentChange.Type.ADDED) {
                
                // Convert it to our Message object
                Message newMessage = dc.getDocument().toObject(Message.class);
                
                // Add to our UI list
                chatList.add(newMessage);
                
                // Tell the RecyclerView to insert the new row and scroll down instantly!
                adapter.notifyItemInserted(chatList.size() - 1);
                recyclerView.scrollToPosition(chatList.size() - 1);
            }
        }
    });
```

### How the Listener Works:
1. When the Activity opens, the listener fires immediately, downloading the entire chat history (e.g., 50 messages) as `ADDED` types.
2. The listener stays open in the background via a persistent WebSocket connection.
3. If the other user sends a message, Firebase instantly pushes that *single new document* down the socket.
4. The listener fires again, but this time `getDocumentChanges()` only contains 1 item (the new message). The UI updates in milliseconds, creating a seamless real-time chat experience!
